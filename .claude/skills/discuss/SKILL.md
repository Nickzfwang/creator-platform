# /discuss — 需求討論與分析

透過結構化提問釐清功能需求，輸出需求摘要。

## 使用方式
```
/discuss $ARGUMENTS
```
其中 $ARGUMENTS 為功能主題或需求描述，例如：會員等級系統, 改善影片上傳流程

## 執行步驟

### 1. 背景調研
- 讀取 `docs/PRODUCT_SPEC.md` 了解現有產品功能
- 讀取 `docs/SYSTEM_ARCHITECTURE.md` 了解技術架構
- 讀取 `docs/DEVELOPMENT_PLAN.md` 了解開發進度
- 搜尋 `apps/api/src/modules/` 和 `apps/web/src/app/` 確認相關模組現狀

### 2. 結構化提問（逐步進行，每次 1-2 個問題）

**Round 1 — 問題定義**
- 這個功能要解決什麼具體問題？
- 目標用戶是誰？（創作者 / 粉絲 / 品牌方 / 管理員）

**Round 2 — 功能範圍**
- 核心功能有哪些？（列出後請使用者確認）
- 明確不包含的功能？（避免 scope creep）
- MVP 版本 vs 完整版本的差異？

**Round 3 — 用戶故事**
- 引導使用者描述 3-5 個核心 user story
- 格式：「作為 {角色}，我想要 {功能}，以便 {目的}」
- 為每個 story 追問驗收條件

**Round 4 — 技術考量**
- 與現有哪些模組有關聯？（基於 Round 1 的調研主動提出）
- 有無特定技術偏好或限制？
- 效能需求？（預估資料量、併發數）
- 第三方服務整合需求？

**Round 5 — 優先級與排程**
- 功能優先級（P0 必須 / P1 重要 / P2 nice-to-have）
- 是否有時間壓力或外部依賴？

### 3. 批判性分析
- 對使用者的需求提出潛在問題或風險
- 如果需求與現有架構衝突，明確指出
- 提出可能被忽略的邊界條件
- **不盲目附和**，有疑慮時主動提出（遵循 CLAUDE.md 協作原則）

### 4. 輸出需求摘要

產出格式：
```markdown
# {功能名稱} — 需求摘要

## 問題陳述
{一段話描述要解決的問題}

## 目標用戶
- {角色 1}: {需求描述}

## 核心 User Stories
1. 作為 {角色}，我想要 {功能}，以便 {目的}
   - 驗收條件：{AC1}, {AC2}

## 功能範圍
### 包含（In Scope）
- {feature 1}

### 不包含（Out of Scope）
- {excluded 1}

## 技術考量
- 相關模組：{modules}
- 外部服務：{services}
- 效能需求：{requirements}

## 風險與待確認項目
- {risk 1}

## 優先級：{P0/P1/P2}
```

### 5. 確認與銜接
- 請使用者 review 需求摘要
- 確認後提示：「需求已確認，你可以執行 `/prd {feature-name}` 產出產品規格文檔」

## 原則
- 每次只問 1-2 個問題，避免一次丟太多問題
- 主動提供選項和建議，降低使用者回答負擔
- 基於已讀取的現有代碼提供具體建議，而非泛泛而談
- 遵循 CLAUDE.md 的批判性協作原則
