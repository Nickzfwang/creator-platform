# 趨勢雷達 2.0 — 產品需求文檔 (PRD)

> Status: draft | Owner: Nick | Priority: P0
> Created: 2026-03-30 | Target: TBD

## 1. 背景與目標

### 問題陳述

現有趨勢雷達（v1）提供 8 個 RSS 來源的 AI 分析，但存在以下問題：

- **來源不足**：缺少 YouTube Trending、TikTok、Threads 等主流創作者平台的即時趨勢
- **無歷史追蹤**：資料為 stateless in-memory cache（30 分鐘 TTL），無法觀察趨勢的生命週期（出現 → 高峰 → 衰退）
- **被動查看**：創作者必須主動進入頁面才能看到趨勢，容易錯過最佳切入時機
- **缺乏個人化**：所有創作者看到一樣的趨勢，無法根據自己的 niche 設定關注焦點
- **Playwright 體驗差**：auto-browse 模組使用 headed mode 開啟瀏覽器視窗，使用者體驗怪異

**核心痛點：創作者需要一個「趨勢雷達」主動偵測並通知與自己相關的趨勢，而非每天手動刷新查看。**

### 目標

- 擴充資料來源至 YouTube Trending、TikTok、Threads、Dcard 等多平台
- 持久化趨勢數據，提供 14 天歷史走勢分析
- 支援關鍵字監控，主動推送趨勢命中通知
- 爆紅偵測演算法，自動辨識熱度突增的主題
- BullMQ 排程自動刷新，Playwright 改為 headless mode 於背景 worker 執行

### 成功指標

| 指標 | 目前值 | 目標值 |
|------|--------|--------|
| 趨勢資料來源數 | 8（RSS only） | 12+（RSS + 平台 API + Playwright） |
| 創作者趨勢頁面停留時間 | 未追蹤 | > 3 分鐘/次 |
| 關鍵字命中通知後的點擊率 | N/A | > 50% |
| 趨勢 → 內容產出轉換率 | N/A | > 20%（看到趨勢後 48 小時內排入日曆） |
| 爆紅偵測提前量 | N/A | 比創作者自行發現早 6-12 小時 |

## 2. 用戶故事與驗收條件

### Story 1: 多平台趨勢總覽

**作為** 內容創作者，**我想要** 同時看到 YouTube、TikTok、Threads、Dcard 及科技媒體的趨勢，**以便** 找到跨平台共同熱點，判斷哪些主題值得投入。

**驗收條件 (Acceptance Criteria):**
- [ ] AC1: Given 系統已完成自動刷新, When 創作者進入趨勢頁面, Then 顯示來自至少 4 個平台類別的趨勢主題，每個主題標註來源平台
- [ ] AC2: Given 趨勢列表已載入, When 創作者篩選特定平台（如 YouTube）, Then 僅顯示該平台相關趨勢
- [ ] AC3: Given 同一主題出現在多個平台, When AI 分析完成, Then 標記為「跨平台熱點」並提升排序權重
- [ ] AC4: Given 趨勢資料正在背景刷新, When 創作者在頁面上, Then 不會看到瀏覽器視窗開啟（headless worker 執行）

### Story 2: 關鍵字監控與通知

**作為** 內容創作者，**我想要** 設定追蹤關鍵字（如「AI 工具」「短影音趨勢」），當有相關趨勢出現時收到通知，**以便** 不用反覆手動查看就能掌握與我相關的動態。

**驗收條件 (Acceptance Criteria):**
- [ ] AC1: Given 創作者在趨勢設定頁, When 新增追蹤關鍵字, Then 關鍵字儲存成功，最多可設定 20 個關鍵字
- [ ] AC2: Given 已設定關鍵字「AI 工具」, When 新趨勢包含「AI 工具」相關內容, Then AI 判斷相關性（非純粹字串比對），並觸發站內通知
- [ ] AC3: Given 關鍵字命中通知已送達, When 創作者點擊通知, Then 跳轉至該趨勢的詳細頁面
- [ ] AC4: Given 創作者啟用了 Email 通知, When 關鍵字命中, Then 同步發送 Brevo Email 通知（可在設定中開關）

### Story 3: 趨勢走勢追蹤

**作為** 內容創作者，**我想要** 看到一個主題在過去 14 天的熱度變化曲線，**以便** 判斷現在切入是否太晚、趨勢是上升還是衰退。

**驗收條件 (Acceptance Criteria):**
- [ ] AC1: Given 趨勢主題有歷史數據, When 創作者點擊該主題, Then 顯示 14 天熱度走勢圖（X 軸：日期, Y 軸：relevanceScore）
- [ ] AC2: Given 走勢圖已顯示, When 趨勢正在上升階段, Then 標記為 🔥「上升中」；若已過高峰, Then 標記為 📉「衰退中」
- [ ] AC3: Given 某主題首次出現, When 查看詳情, Then 顯示「新趨勢」標記及首次偵測時間
- [ ] AC4: Given 歷史數據超過 14 天, When 系統執行清理 cron, Then 自動刪除過期快照，保持資料量可控

### Story 4: 每日趨勢摘要 Email

**作為** 內容創作者，**我想要** 每天早上收到一封趨勢摘要信，**以便** 打開信箱就能快速掌握今日值得做的主題，不用登入平台。

**驗收條件 (Acceptance Criteria):**
- [ ] AC1: Given 創作者啟用了每日摘要, When 每天早上 9:00, Then 系統透過 Brevo 發送趨勢摘要 Email
- [ ] AC2: Given Email 已送達, When 創作者開啟信件, Then 看到：今日 Top 5 趨勢、每個趨勢的一句話摘要、與其追蹤關鍵字的命中提示、「查看完整報告」的 CTA 連結
- [ ] AC3: Given 創作者不想收到每日摘要, When 在設定中關閉, Then 停止發送，不影響站內通知
- [ ] AC4: Given Email 使用 Brevo 發送, When 查看 Email, Then 使用平台品牌模板，包含取消訂閱連結（符合 CAN-SPAM）

### Story 5: 爆紅偵測與即時提醒

**作為** 內容創作者，**我想要** 在某個主題的熱度突然飆升時立即收到提醒，**以便** 搶先製作相關內容，抓住流量紅利。

**驗收條件 (Acceptance Criteria):**
- [ ] AC1: Given 系統完成一輪趨勢刷新, When 某主題的 relevanceScore 較前次上升 ≥ 0.3 或首次出現且 score ≥ 0.8, Then 觸發爆紅偵測
- [ ] AC2: Given 爆紅偵測觸發, When 創作者在線上, Then 即時站內通知（WebSocket push），標記為「🔥 爆紅警報」
- [ ] AC3: Given 爆紅通知已送達, When 創作者查看, Then 顯示：主題摘要、熱度飆升幅度、推薦內容切入角度（AI 生成）、相關原始來源連結
- [ ] AC4: Given 創作者啟用了 Email 通知, When 爆紅偵測觸發, Then 同步發送 Brevo Email（不等待每日摘要排程）

## 3. 功能範圍

### 包含（In Scope）

| 功能 | 說明 |
|------|------|
| 多平台資料源擴充 | YouTube Trending (RSS)、TikTok (Playwright headless)、Threads (Playwright headless)、Dcard (JSON API，維持現有) |
| 趨勢持久化 | 新增 Prisma models，每次刷新儲存 snapshot，保留 14 天 |
| 關鍵字監控 | 使用者可設定追蹤關鍵字，AI 語意比對（非純字串匹配） |
| 站內通知系統 | 新增 Notification model + WebSocket 即時推送，供全平台模組共用 |
| Brevo Email 通知 | 每日趨勢摘要 + 關鍵字/爆紅即時通知 |
| 爆紅偵測 | 基於 relevanceScore 差值的突增偵測演算法 |
| 趨勢走勢圖 | 14 天歷史曲線 + 趨勢階段標記（上升/高峰/衰退） |
| BullMQ 排程 | RSS/API 每 2 小時白天刷新、Playwright 每 6 小時、14 天資料清理 cron |
| Playwright 改善 | 改為 headless mode + stealth plugin，移至 BullMQ worker 執行 |

### 不包含（Out of Scope）

| 功能 | 原因 |
|------|------|
| 推播通知（FCM/APNs） | 需要 service worker + 行動端，投入產出比低，P2 再評估 |
| Chrome 擴充整合 | 獨立模組，不在此次迭代 |
| 自訂 RSS 來源 | 使用者可能加入低品質來源，影響 AI 分析品質，未來再開放 |
| 情緒分析（sentiment） | 可作為未來增強，非核心需求 |
| 即時串流（SSE/WebSocket 趨勢更新） | 每 2 小時刷新足夠，即時串流過度設計 |

## 5. 數據指標與追蹤

### 5.1 核心指標

| 指標 | 追蹤方式 | 目的 |
|------|----------|------|
| 趨勢頁面日活（DAU） | 前端 page view | 衡量功能使用頻率 |
| 關鍵字設定數/用戶 | DB 統計 | 衡量個人化採用率 |
| 通知點擊率（CTR） | 通知 readAt 時間差 | 衡量通知品質 |
| 趨勢 → 日曆轉換率 | 「排入日曆」按鈕事件 | 衡量行動轉換 |
| 爆紅偵測準確率 | 人工抽查 | 衡量演算法品質 |
| Email 開信率 | Brevo webhook | 衡量 Email 品質 |

### 5.2 資料保留策略

| 資料類型 | 保留期限 | 清理方式 |
|----------|----------|----------|
| 趨勢快照（TrendSnapshot + TrendTopic） | 14 天 | 每日凌晨 cron 刪除 |
| 通知記錄 | 30 天 | 每日凌晨 cron 刪除 |
| 關鍵字設定 | 永久 | 使用者手動刪除 |
| Email 發送記錄 | 90 天 | Brevo 端管理 |

## 6. 技術約束與依賴

### 6.1 外部依賴

| 服務 | 用途 | 費用考量 |
|------|------|----------|
| OpenAI GPT-4o-mini | 趨勢分析、語意比對、摘要生成 | 每次刷新約 ~$0.01（50 items） |
| Brevo | Email 發送 | 免費方案 300 封/天，足夠初期 |
| YouTube RSS | Trending 資料 | 免費，無 API key |
| Dcard API | 熱門貼文 | 公開 API，無需認證 |
| Playwright | TikTok/Threads 爬取 | 免費，需管理 browser 資源 |

### 6.2 技術決策

| 決策 | 選擇 | 理由 |
|------|------|------|
| 趨勢匹配方式 | AI 語意比對（非字串 match） | 避免漏掉語意相關但用詞不同的趨勢 |
| 通知推送 | WebSocket (socket.io) | 專案已有依賴，即時性高 |
| Email 服務商 | Brevo | 使用者已有帳號，免費額度足夠 |
| Playwright mode | Headless + stealth plugin | 解決 headed mode 體驗問題 |
| 爆紅偵測 | Score 差值 ≥ 0.3 或首次 ≥ 0.8 | 簡單有效，可後續調參 |
| 歷史保留 | 14 天 | 涵蓋完整趨勢生命週期，資料量可控 |
| 刷新排程 | RSS 每 2hr（白天）/ Playwright 每 6hr | 平衡即時性與資源消耗 |

### 6.3 與現有模組的關聯

| 模組 | 整合方式 |
|------|----------|
| Content Strategy | 已整合：`getTrends()` 被 AI 主題推薦引用，維持不變 |
| Notification（新增） | 全平台共用通知基礎設施，趨勢雷達為首個使用者 |
| Email Marketing | 共用 Brevo 基礎設施，但獨立管理趨勢 Email 模板 |
| Auto-Browse | 重構：Playwright 爬蟲改為 headless，由 BullMQ worker 統一調度 |
| Post Scheduler | 未來可擴展：趨勢 → 排入日曆 → 自動排程發佈 |

## 7. 風險與緩解

| 風險 | 影響 | 緩解措施 |
|------|------|----------|
| Playwright 被平台封鎖 | TikTok/Threads 資料中斷 | stealth plugin + 隨機 UA + 限制頻率；封鎖時 graceful fallback 到 RSS |
| OpenAI API 故障 | 無法分析趨勢 | 返回原始 feed 資料（無 AI 分析），降級但不中斷 |
| 通知過多造成疲勞 | 使用者關閉所有通知 | 預設保守閾值 + 每日摘要合併 + 使用者可調整敏感度 |
| Brevo 免費額度用盡 | Email 發送失敗 | 監控用量，接近上限時降低發送頻率或提示升級 |
| 爆紅偵測誤報 | 降低信任度 | 初期閾值偏嚴格（score 差值 ≥ 0.3），根據回饋調整 |
