# AI Coding Workflow — 命令說明與開發流程

> 本文檔說明 Creator Platform 的 AI 輔助開發流程。所有命令在 Claude Code CLI 中以 `/` 前綴觸發。

---

## 流程總覽

```
┌─────────────────────────────────────────────────────────────────┐
│                    /feature {功能名稱}                            │
│              （總指揮 — 自動引導走完以下所有階段）                    │
└──────┬──────────────────────────────────────────────────────────┘
       │
       ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│  1. /discuss  │───▶│   2. /prd    │───▶│   3. /spec   │
│   需求討論    │    │  產品規格     │    │  技術規格     │
└──────────────┘    └──────────────┘    └──────────────┘
                                               │
       ┌───────────────────────────────────────┘
       ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│ 4. /implement │───▶│  5. /review  │───▶│   6. /test   │───▶│   7. /e2e    │
│    實作       │    │  代碼審查     │    │  單元測試     │    │  E2E 測試    │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
```

---

## 使用方式

### 方式一：總指揮模式（推薦用於新功能）

```bash
/feature 會員等級系統
```

自動引導走完 7 個階段，每個階段結束後暫停等你確認。你可以：
- 說「繼續」→ 進入下一階段
- 說「跳過」→ 略過當前階段
- 說「從 Phase 5 開始」→ 跳到指定階段
- 說「快速模式」→ Phase 5-7 自動執行不暫停

### 方式二：單獨執行（推薦用於補充或局部工作）

每個命令都可以獨立使用，不需要走完整流程：

```bash
/discuss 改善影片上傳體驗     # 只想討論需求
/prd membership-tier          # 只產 PRD
/spec payment                 # 只產技術規格
/implement auth               # 只實作模組
/review                       # Review 當前變更
/test user                    # 只跑單元測試
/e2e auth                     # 只跑 E2E 測試
```

## 各命令詳細說明

### `/discuss` — 需求討論與分析

| 項目 | 說明 |
|------|------|
| **用途** | 透過結構化提問釐清功能需求 |
| **輸入** | 功能主題或需求描述 |
| **輸出** | 需求摘要（可選存為 `specs/discuss-{name}.md`） |
| **適用時機** | 新功能構想初期、需求不明確時 |

**流程**：
1. 自動讀取現有產品文檔和架構，了解專案現狀
2. 分 5 輪提問：問題定義 → 功能範圍 → 用戶故事 → 技術考量 → 優先級
3. 每輪 1-2 個問題，不會一次丟太多
4. 主動提出潛在風險和衝突（不盲目附和）
5. 產出結構化需求摘要

**範例**：
```
/discuss 品牌方自助投放廣告
→ 討論目標用戶、核心流程、與現有 brand-deal 模組的關係
→ 輸出需求摘要含 user stories 和驗收條件
```

---

### `/prd` — 產品需求文檔

| 項目 | 說明 |
|------|------|
| **用途** | 產出面向產品的 PRD 文檔 |
| **輸入** | 功能名稱（會自動讀取 discuss 摘要） |
| **輸出** | `specs/prd-{name}.md` |
| **適用時機** | 需求確認後，開始技術設計前 |

**包含內容**：
- 背景與目標 + 成功指標
- 用戶故事 + **可測試的驗收條件**（Given/When/Then）
- 功能規格 + 業務規則
- UI/UX 流程
- 範圍定義（In/Out of Scope）
- 風險與里程碑

**與 `/spec` 的分工**：

| | `/prd` | `/spec` |
|--|--------|---------|
| 讀者 | PM、設計師、利害關係人 | 工程師 |
| 重點 | What & Why | How |
| 內容 | 用戶故事、驗收條件、業務規則 | API 設計、資料模型、技術細節 |

---

### `/spec` — 技術規格文檔

| 項目 | 說明 |
|------|------|
| **用途** | 產出面向工程師的技術規格 |
| **輸入** | 模組名稱 |
| **輸出** | `specs/{phase}-{name}.md` |
| **適用時機** | PRD 確認後，開始實作前 |

**包含內容**：
- 資料模型（引用 Prisma schema）
- API Endpoints（完整 TypeScript request/response 型別）
- Business Logic（流程步驟 + 邊界條件）
- 前端頁面需求（路由、元件、狀態管理）
- 測試案例清單（Unit + E2E，對應 PRD 驗收條件）

**自動銜接**：如有 PRD，會逐條比對驗收條件確保技術 spec 無遺漏。

---

### `/implement` — 模組實作

| 項目 | 說明 |
|------|------|
| **用途** | 根據技術 spec 實作模組代碼 |
| **輸入** | 模組名稱 |
| **輸出** | 後端模組 + 前端頁面（依 spec 定義） |
| **適用時機** | 技術 spec 確認後 |

**實作順序**：
1. Prisma Schema 變更 → `db:push` + `db:generate`
2. 後端：DTOs → Service → Controller → Module
3. 前端：API hooks → 頁面元件 → 表單驗證 → 狀態管理
4. 品質確認：lint + TypeScript build check

**前置檢查**：自動確認依賴模組是否已實作，未滿足時警告。

---

### `/review` — 代碼審查

| 項目 | 說明 |
|------|------|
| **用途** | 檢查代碼品質、安全性、規範合規性 |
| **輸入** | 無（檢查 git changes）或指定路徑 |
| **輸出** | Review 報告（Critical / Warning / Info） |
| **適用時機** | 實作完成後、commit 前 |

**檢查維度**（7 大類）：
1. **命名規範** — kebab-case 檔案、PascalCase 類別
2. **NestJS 規範** — 模組結構、Swagger 裝飾器
3. **安全性** — OWASP Top 10、token 加密、無硬編碼 secret
4. **多租戶** — tenantId 過濾、無跨租戶洩漏
5. **型別安全** — 無 any、完整 DTO 型別
6. **錯誤處理** — NestJS exceptions、有意義的錯誤訊息
7. **效能** — N+1 查詢、cursor 分頁、select 優化

**自動修復**：發現問題後會詢問是否自動修復。

---

### `/test` — 單元測試

| 項目 | 說明 |
|------|------|
| **用途** | 生成並執行模組的單元測試 |
| **輸入** | 模組名稱 |
| **輸出** | `__tests__/*.spec.ts` 測試檔案 + 執行結果 |
| **適用時機** | 實作完成且 review 通過後 |

**測試範圍**：
- Service 單元測試（業務邏輯）
- Controller 單元測試（HTTP 層）
- 每個 public method: happy path + error case
- Mock 外部依賴（Prisma, 外部 API, BullMQ）

**執行**：`pnpm --filter api test -- --testPathPattern={module}`

---

### `/e2e` — E2E 測試

| 項目 | 說明 |
|------|------|
| **用途** | 基於 PRD 驗收條件生成 Playwright E2E 測試 |
| **輸入** | 模組名稱 |
| **輸出** | `apps/web/e2e/*.spec.ts` + 執行結果 |
| **適用時機** | 單元測試通過後、功能完成的最終驗證 |

**特點**：
- 測試名稱以 `AC{N}:` 開頭，直接對應 PRD 驗收條件
- 使用 Playwright 最佳實踐（getByRole, getByText 優先）
- 包含登入等共用 fixtures
- 輸出涵蓋率報告（PRD 驗收條件覆蓋百分比）

**首次使用**：會自動檢查並引導安裝 Playwright 環境。

## 文件產出對照表

| 階段 | 產出文件 | 位置 |
|------|---------|------|
| /discuss | 需求摘要 | (對話中，可選存為 `specs/discuss-{name}.md`) |
| /prd | 產品需求文檔 | `specs/prd-{name}.md` |
| /spec | 技術規格文檔 | `specs/{phase}-{name}.md` |
| /implement | 模組代碼 | `apps/api/src/modules/{name}/` + `apps/web/src/` |
| /review | Review 報告 | (對話中) |
| /test | 單元測試 | `apps/api/src/modules/{name}/__tests__/` |
| /e2e | E2E 測試 | `apps/web/e2e/{name}.spec.ts` |

## 常見場景指南

### 全新功能開發
```bash
/feature 會員等級系統
# 自動走完 discuss → prd → spec → implement → review → test → e2e
```

### 現有模組補測試
```bash
/test auth
/e2e auth
```

### 只想討論一個想法
```bash
/discuss AI 自動剪輯推薦
```

### 快速修 Bug（不需要完整流程）
直接描述問題，不用任何命令，Claude 會直接協助修復。

### 補產品文檔
```bash
/prd video-upload
```

### 重構前先 Review
```bash
/review apps/api/src/modules/video/
```

## 品質保證機制

```
Code 品質 ← /review（7 維度檢查）
    │
    ├── 安全性 ← OWASP Top 10 + 多租戶隔離
    ├── 規範性 ← CLAUDE.md 開發規範
    ├── 正確性 ← /test（單元測試）
    └── 完整性 ← /e2e（驗收條件覆蓋率）
```

每個環節都可以獨立執行，但 `/feature` 會自動串接確保不遺漏。
